# Лабораторна робота №1: Розгортання Web-сервісу з автоматизацією

**Призначення застосунку:** Notes Service — це простий сервіс для зберігання текстових нотаток. Він дозволяє створювати нові записи (з полями `id`, `title`, `content`, `created_at`), переглядати загальний список та читати деталі конкретної нотатки.

Проєкт реалізує просту веб-платформу на віртуальній машині Linux з автоматизацією встановлення та налаштування, а також обмеженим адмініструванням сервісів.

## Індивідуальне завдання - розрахунок варіанту (Порядковий номер N = 18)

- V2 = (18 % 2) + 1 = 0 + 1 = **1** → Конфігурація через аргументи командного рядка + MariaDB
- V3 = (18 % 3) + 1 = 0 + 1 = **1** → Notes Service
- V5 = (18 % 5) + 1 = 3 + 1 = **4** → Порт застосунку 8000

## Архітектура системи

`client → nginx (reverse proxy) → web application (Node.js/Express) → MariaDB`

- **Nginx** слухає на `0.0.0.0:80`.
- **Web App** слухає на `127.0.0.1:8000` (через Systemd Socket Activation).
- **MariaDB** слухає на `127.0.0.1:3306`.

- Доступ до бази даних обмежено лише з локальної машини (`localhost`).
- Клієнти взаємодіють виключно через Nginx (reverse proxy).
- Web-застосунок запущено через systemd з socket activation.

## Локальна розробка (Development)

Для того, щоб запустити застосунок локально без скрипта розгортання:

1. Встановіть Node.js (v20+) та MariaDB.
2. Створіть базу даних та користувача:
   ```sql
   CREATE DATABASE notes_db;
   CREATE USER 'app'@'127.0.0.1' IDENTIFIED BY '12345678';
   GRANT ALL PRIVILEGES ON notes_db.* TO 'app'@'127.0.0.1';
   ```
3. Встановіть залежності: `npm install`
4. Запустіть міграцію: `node migrate.js --db-user=ваш_юзер --db-pass=ваш_пароль --db-name=notes_db`
5. Запустіть сервер: `node server.js --db-user=ваш_юзер --db-pass=ваш_пароль --db-name=notes_db --port=8000`

## Автоматичне розгортання

Для розгортання виконайте наступні команди на чистій ОС:

```
git clone https://github.com/DenysP21/kpi-lab1-deployment
cd kpi-lab1-deployment
chmod +x deploy.sh
sudo ./deploy.sh
```

**Що робить скрипт:**

1. Встановлює необхідні пакети (Node.js 20, Nginx, MariaDB, git, sudo тощо)
2. Створює користувачів: student, teacher, operator, app
3. Налаштовує MariaDB: створює БД notes_db, користувача app та надає права
4. Копіює код у /opt/mywebapp, встановлює залежності через npm ci --production
5. Створює systemd socket та service (mywebapp.socket, mywebapp.service)
6. Налаштовує Nginx як reverse proxy (слухає 80, проксі на 8000, блокує /health/)
7. Створює файл /home/student/gradebook з числом 18
8. Блокує дефолтного користувача (UID 1000), якщо він не student

Після виконання система повністю готова до роботи.

## Користувачі та права

- `student` / `teacher`: Адміністративні права (`sudo`). Пароль: `12345678` (потребує зміни при першому вході).
- `app`: Системний користувач для роботи застосунку (без доступу до shell).
- `operator`: Обмежений доступ. Може керувати сервісом `mywebapp.service` та перевантажувати Nginx. Пароль: `12345678`.

## API Ендпоінти

- `GET /` — головна сторінка з переліком ендпоінтів (тільки HTML).
- `GET /notes` — Отримати список усіх нотаток.
- `POST /notes` — Створити нову нотатку.
  - Очікує тіло запиту: `{"title": "Назва", "content": "Текст"}`
  - Повертає `201 Created` (якщо Accept: application/json) або робить редирект (якщо Accept: text/html).
- `GET /notes/<id>` — Отримати детальну інформацію про нотатку за її ID.
- `GET /health/alive` — Перевірка стану сервера (повертає 200 OK). Доступ ззовні закрито через Nginx (згідно з вимогою експонувати виключно бізнес-логіку).
- `GET /health/ready` — Перевірка підключення до БД (повертає 200 OK або 500). Доступ ззовні закрито через Nginx.

### Content Negotiation:

- Якщо Accept: text/html → повертається проста HTML-сторінка (таблиці, без JS/CSS)
- Якщо Accept: application/json (або відсутній) → JSON

## Інструкція тестування розгорнутої системи

_Примітка:_ У командах нижче замініть <IP-адреса-ВМ> на:

- 127.0.0.1, якщо ви тестуєте безпосередньо всередині віртуальної машини (через консоль або SSH).
- IP-адресу віртуальної машини (або 127.0.0.1:8080 при NAT Port Forwarding), якщо ви тестуєте з основної ОС.

Після запуску deploy.sh:

```
# 1. Перевірка створення користувачів (має вивести student, teacher, operator та app)
grep -E 'student|teacher|operator|app' /etc/passwd

# 2. Перевірка головної сторінки
curl http://<IP-адреса-ВМ>/

# 3. Створення нотатки
curl -X POST -H "Content-Type: application/json" \
 -d '{"title":"Тестова нотатка","content":"Це тест"}' \
 http://<IP-адреса-ВМ>/notes

# 4. Перевірка списку нотаток (HTML)
curl http://<IP-адреса-ВМ>/notes

# 5. Перевірка списку нотаток (JSON)
curl -H "Accept: application/json" http://<IP-адреса-ВМ>/notes

# 6. Перевірка детальної нотатки
curl http://<IP-адреса-ВМ>/notes/1

# 7. Перевірка, що health закрито ззовні
curl -I http://<IP-адреса-ВМ>/health/ready # має повернути 403

# 8. Перевірка прав operator
# Спочатку залогіньтесь як operator:
su - operator
# Пароль: 12345678 (або той, який ви встановили після першого входу)
# Спробуйте виконати дозволену команду:
sudo systemctl status mywebapp.service
# Спробуйте виконати заборонену команду (має бути відмова):
sudo systemctl stop sshd

# 9. Перевірка Systemd Socket Activation
# Зупиняємо основний сервіс (але залишаємо сокет активним)
sudo systemctl stop mywebapp.service
# Перевіряємо статус - має бути "inactive (dead)"
systemctl status mywebapp.service
# Робимо запит до Nginx (який смикне сокет)
curl http://<IP-адреса-ВМ>/
# Знову перевіряємо статус - сервіс мав автоматично "прокинутись" і стати "active (running)"
systemctl status mywebapp.service
```
